# Query History Search Performance Spec

Date: 2026-06-15
Priority: P6
Area: Backend query history and search
Primary files: `engine/api/query.py`, query history model/table definitions

## Problem

Query history search performs `ILIKE '%term%'` across multiple fields. This does not use normal B-tree indexes and becomes slow as history grows. The same issue affects Chinese full fuzzy matching, where substring search is both expensive and often low quality.

## Goals

- Keep simple search behavior for small local datasets.
- Add a scalable path for larger query history.
- Improve Chinese search behavior without introducing a heavy external service.
- Keep API response shape stable.

## Non-Goals

- Do not introduce Elasticsearch or a remote search service.
- Do not build cross-workspace semantic search in this iteration.
- Do not change how SQL text is redacted before persistence.

## Proposed Design

Use a two-tier search strategy.

Tier 1: bounded simple search.

- Keep current `LIKE`/`ILIKE` behavior for small result sets.
- Add a hard `limit` and default sort by recent update time.
- Restrict default search fields to the highest-value fields: title, question, SQL preview.

Tier 2: local full-text search.

- For SQLite-backed local history, add an FTS5 virtual table when SQLite supports it.
- Index redacted query title/question/sql text.
- Use tokenizer appropriate for available SQLite build. If Chinese tokenizer support is unavailable, fall back to trigram-like normalized keyword tokens generated by application code.
- Keep synchronization simple: update FTS row when query history row is created, updated, or deleted.

API behavior:

- Existing endpoint accepts the same `search` parameter.
- Internally choose FTS when available and history count crosses a threshold, otherwise use bounded simple search.
- Include no new response fields in first version.

## Acceptance Criteria

- Search over large local history does not scan unbounded rows by default.
- SQLite FTS path is covered by tests when available.
- Fallback path works when FTS5 is unavailable.
- Chinese terms are normalized consistently before indexing/search.
- Existing query history API consumers do not need changes.

## Test Plan

- Unit test search field selection and limit behavior.
- Integration test with many synthetic history rows verifies bounded result count.
- FTS-enabled test verifies exact and partial keyword lookup.
- Chinese search test with representative terms verifies expected rows are found.

## Rollout

Ship behind an internal capability check. Log whether query history search is using FTS or fallback simple search.
