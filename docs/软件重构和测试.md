# DBFox 软件质量深度分析报告

## 一、项目整体理解

DBFox 是一款**本地优先、AI 原生的数据库工作台桌面应用**。  
技术栈：Python 3.12 + FastAPI + SQLAlchemy（后端引擎）/ React 19 + TypeScript + Vite + Zustand（前端）/ Tauri 2 Rust（桌面壳）。  
通信方式：HTTP REST + SSE 流式推送，本地 `127.0.0.1:18625`，通过 `X-Local-Token` 认证。

### 核心用户流程

1. 配置数据源（MySQL / PostgreSQL / SQLite，支持 SSH 隧道 + SSL）
2. 同步 Schema → 浏览表结构 → 查看 ER 图
3. SQL 控制台执行查询（Monaco 编辑器）
4. AI 问数 Agent（自然语言 → SQL → 安全校验 → 执行 → 可视化结果）

### 最高优先级模块（出错影响数据/安全/业务状态）

| 优先级 | 模块 | 出错影响 |
|--------|------|----------|
| P0 | `engine/sql/guardrail.py` | SQL 注入直达生产数据库 |
| P0 | `engine/crypto.py` | 数据库密码泄露 |
| P0 | `engine/main.py` 中间件 | Token 绕过导致未授权访问 |
| P0 | `engine/sql/executor.py` | 错误执行写操作或数据泄露 |
| P1 | `engine/agent_core/persistence.py` | Agent 运行状态丢失/不一致 |
| P1 | `engine/datasource.py` | SSH 隧道泄漏、连接池膨胀 |
| P1 | `engine/api/agent.py` | 事务未回滚导致数据库状态腐化 |

## 三、主要质量优点

1. **安全模型设计精良**：Guardrail（AST 级 SQL 静态分析）+ TrustGate（运行时安全决策）+ 数据脱敏 + 确认机制 + Origin 校验 + `secrets.compare_digest()` Token 鉴权，形成多层纵深防御。
2. **加密存储设计合理**：AES-256-GCM 加密，密钥优先存储 OS Keychain，有完整迁移路径（文件→Keychain）。
3. **类型系统完备**：Pydantic v2 + TypeScript 严格类型，API 契约清晰。
4. **错误层次清晰**：`DBFoxError` 基类 + 6 个语义子类，全局兜底处理器，不泄露调用栈。
5. **测试策略有深度**：后端 54 个测试文件，含架构守卫测试、安全测试（30+ guardrail 用例）、信任门测试。
6. **工具系统可扩展**：YAML 声明式工具定义 + Handler 注册 + 优先级覆盖，支持用户扩展。
7. **连接池管理**：MySQL/PostgreSQL 使用 SQLAlchemy QueuePool，带 LRU 驱逐和健康检查。
8. **持久化 Sink 分层**：`Noop`/`Session`/`Sync` 策略模式，eval 与生产模式清晰隔离。

## 四、高风险问题清单

### 🔴 严重（Critical）

**C1. EXPLAIN 语句使用 f-string 拼接**
- 位置：`engine/sql/executor.py` L309 (SQLite), L345 (MySQL)
- 代码：`cursor.execute(f"EXPLAIN QUERY PLAN {safe_sql}")`
- 风险：若 guardrail 存在绕过漏洞，此处直接成为二次注入点。
- 修复：在执行前对 `safe_sql` 进行再次单语句 + 无分号校验。

**C2. `SyncPersistenceSink.record_artifact` 静默丢弃（数据丢失）**
- 位置：`engine/agent_core/persistence_sink.py` L144
- 代码：`pass  # for now`
- 风险：Agent 所有 Artifact（SQL 结果、图表）在默认生产模式下被静默丢弃，前端看不到任何结果卡片。
- 修复：实现 `self._write(lambda db: ap.record_artifact(db, ...))`。

**C3. `api_test_connection` 引用未 import 的 `DataSourceConnectionError`**
- 位置：`engine/api/datasources.py` L137
- 代码：`raise DataSourceConnectionError(...)` 但文件顶部未 import 该类。
- 风险：连接测试失败时触发 `NameError`，FastAPI 返回 500 而非 400，用户看到内部错误。
- 修复：在 import 块添加 `from engine.errors import DataSourceConnectionError`。

### 🟡 高（High）

**H1. PostgreSQL 数据源 EXPLAIN 使用 MySQL 连接池**
- 位置：`engine/sql/executor.py` L336 `else:` 分支
- 问题：`explain_sql` 的方言分支只有 `sqlite` 和 `else(mysql)`，PostgreSQL 走 else 调用 MySQL 驱动。
- 修复：添加 `elif db_type in ("postgresql", "postgres"):` 分支。

**H2. `_query_stats_for_table` 使用 `contains()` 全表扫描**
- 位置：`engine/tools/db_tools.py` L418-440
- 代码：`QueryHistory.executed_sql.contains(table_name)` → SQLite `LIKE '%name%'` 无法走索引。
- 风险：随 `query_history` 表增长（当前本地 DB 已 185MB）性能严重退化。

**H3. `SyncPersistenceSink` 使用匿名类伪造 Request 对象**
- 位置：`engine/agent_core/persistence_sink.py` L122
- 代码：`type("Req", (), {"datasource_id": datasource_id, ...})()`
- 风险：绕过 Pydantic 验证，`max_steps` 硬编码为 20，若内部访问其他字段将产生 `AttributeError`。

**H4. `list_conversations` 自愈逻辑缺少最终 `db.commit()`**
- 位置：`engine/api/conversations.py` L30-47
- 问题：自愈 sync 的写操作在会话关闭时可能回滚。

**H5. `TTLMemoryCache` 无容量上限**
- 位置：`engine/tools/db_tools.py` L770-792
- 问题：`dict` 只有惰性驱逐，长时间运行后无限增长（潜在内存泄漏）。

### 🟢 中（Medium）

**M1. `explain_sql` 在 PostgreSQL 路径未实现（H1 的直接影响）**

**M2. `_save_trace_events` 静默吞掉异常后不上报**
- 位置：`engine/agent_core/persistence.py` L244
- 问题：trace 事件写入失败只打 warning，排查困难。

**M3. FTS5 查询 token 拼接无白名单过滤**
- 位置：`engine/tools/db_tools.py` L520
- 代码：`fts_query = " OR ".join(exact_parts + token_parts)`
- 问题：恶意 token（如含 FTS5 操作符 `AND NOT`）可改变查询语义。

**M4. ORM 模型大量使用 `type: ignore[assignment]`（约 40 处）**
- 位置：`engine/agent_core/persistence.py` 全文
- 原因：模型未迁移到 SQLAlchemy 2.0 `Mapped[]` 类型注解。

**M5. `_requires_confirmation` 对 auto_limit warn 过于激进**
- 位置：`engine/sql/trust_gate.py` L237
- 问题：`SELECT *`（guardrail 已自动修复为加 LIMIT）触发生产确认弹窗，严重影响 Agent 效率。

## 五、具体缺陷报告

### 缺陷 D1：EXPLAIN f-string 拼接（安全）

| 字段 | 内容 |
|------|------|
| 标题 | `explain_sql` 使用 f-string 拼接执行 EXPLAIN |
| 位置 | `engine/sql/executor.py` L309, L345 |
| 类型 | 安全 |
| 严重程度 | 高 |
| 触发条件 | 用户或 Agent 调用 EXPLAIN 功能 |
| 修复方案 | 执行前验证 `safe_sql` 无分号、无危险函数，或改为字符串拼接后再次 guardrail 校验 |
| 测试用例 | 传入含 `;` 的 safe_sql，验证被拒绝而非执行 |

### 缺陷 D2：`record_artifact` 静默 pass（数据丢失）

| 字段 | 内容 |
|------|------|
| 标题 | `SyncPersistenceSink.record_artifact` 静默丢弃所有 Artifact |
| 位置 | `engine/agent_core/persistence_sink.py` L144 |
| 类型 | 功能缺失 / 数据丢失 |
| 严重程度 | 高 |
| 触发条件 | 任何 Agent 运行产生 Artifact |
| 修复方案 | 实现 `self._write(lambda db: ap.record_artifact(db, session_id, run_id, artifact, index))` |
| 测试用例 | 运行 Agent → 查询 `agent_artifacts` 表 → 验证有记录 |

### 缺陷 D3：`DataSourceConnectionError` NameError

| 字段 | 内容 |
|------|------|
| 标题 | `api_test_connection` 引用未 import 的异常类 |
| 位置 | `engine/api/datasources.py` L137 |
| 类型 | 正确性 / 运行时错误 |
| 严重程度 | 高 |
| 触发条件 | 连接测试失败时抛出非 `DBFoxError` 异常 |
| 实际结果 | `NameError` → FastAPI 返回 500 |
| 修复方案 | 文件顶部添加 `from engine.errors import DataSourceConnectionError` |
| 测试用例 | 用错误密码调用 `/datasources/test`，断言返回 400 非 500 |

### 缺陷 D4：PostgreSQL EXPLAIN 调用 MySQL 连接池

| 字段 | 内容 |
|------|------|
| 标题 | PostgreSQL 数据源的 EXPLAIN 使用 MySQL 驱动 |
| 位置 | `engine/sql/executor.py` L336-368 |
| 类型 | 正确性 |
| 触发条件 | 在 PostgreSQL 数据源点击"执行计划" |
| 修复方案 | 添加 `elif db_type in ("postgresql", "postgres"):` 分支，使用 `get_postgres_pool` |

### 缺陷 D5：TTLMemoryCache 无容量上限

| 字段 | 内容 |
|------|------|
| 标题 | `db.inspect` 缓存无容量限制 |
| 位置 | `engine/tools/db_tools.py` L770 |
| 类型 | 性能 / 内存 |
| 修复方案 | 添加 `maxsize: int = 500` 参数，超出时 LRU 驱逐 |

### 缺陷 D6：`_requires_confirmation` auto_limit warn 触发误拦截

| 字段 | 内容 |
|------|------|
| 标题 | guardrail 已修复的 warn 级别查询误触发生产确认 |
| 位置 | `engine/sql/trust_gate.py` L237 |
| 类型 | 用户体验 |
| 修复方案 | 对 `auto_limit` 和 `select_star`（且 guardrail 已注入 LIMIT）不升级为 `warning` 风险级别 |

## 八、集成测试场景

1. **数据源完整生命周期**：创建 → 连接测试 → Schema 同步 → 浏览表 → SQL 执行 → 删除（含确认流程）

5. **引擎重启恢复**：Agent 运行中 → 重启引擎 → 验证 `AgentRun.status` 被标记为 `interrupted`
