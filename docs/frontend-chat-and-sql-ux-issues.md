# DBFox 前端对话与 SQL 结果展示问题梳理

> 创建日期：2026-06-21  
> 范围：对话页、SQL 执行结果展示、Chart 渲染、对话持久化、设置页 UI 优化

## 背景

当前 DBFox 在对话、SQL 结果展示和部分页面 UI 上存在若干体验与功能问题，影响用户查看 AI 回复、查看多 SQL 执行结果、复用历史对话以及进行设置配置。本文档用于记录问题现象、可能原因、修复建议和验收标准，便于后续拆分 issue 或直接进入开发。

## 问题清单

### 1. Chart 渲染异常

#### 现象

对话或 SQL 结果中生成图表时，Chart 渲染存在异常，可能表现为：

- 图表没有正常显示；
- 图表容器尺寸异常；
- 数据已返回但前端没有正确渲染；
- 切换对话或刷新后图表丢失；
- 多 SQL 场景下图表绑定到了错误的结果集。

#### 影响

用户无法直观看到 SQL 查询结果的可视化展示，降低分析体验。

#### 建议排查方向

- 检查 Chart 组件入参是否稳定，尤其是 `data`、`columns`、`type`、`options` 等字段；
- 检查 SQL 执行结果与 Chart 配置之间的映射关系，避免多结果场景下只使用最后一个结果；
- 检查组件 key 是否唯一，避免 React/Vue 复用旧组件导致渲染错乱；
- 检查异步数据返回后是否正确触发重新渲染；
- 检查空数据、字段类型不匹配、时间字段、数值字段等边界情况。

#### 验收标准

- 单条 SQL 生成 Chart 时可以稳定渲染；
- 多条 SQL 返回多个结果集时，每个 Chart 能绑定到正确的数据源；
- 刷新、切换对话、重新打开历史记录后 Chart 仍能正常显示；
- Chart 空状态、错误状态、加载状态都有合理提示。

### 2. 多 SQL 执行结果只保留最后一条 SQL / SQL Artifact 只显示最后结果

#### 现象

当 AI 回复或用户输入中包含多条 SQL 时，前端结果展示或工件区域似乎只保留了最后一条 SQL 的结果。例如执行 `sql1`、`sql2` 后，最终只看到 `sql2` 对应的 SQL 或结果，前面的 SQL 结果被覆盖或没有展示。

#### 影响

- 用户无法查看完整的多 SQL 执行结果；
- 如果前面的 SQL 是建表、聚合、临时查询或关键分析步骤，会丢失上下文；
- Chart 渲染可能因此只基于最后一个结果集，导致分析不完整。

#### 建议设计

多 SQL 结果应以数组结构保存和展示，每一条 SQL 都应该拥有独立的元信息和执行结果。

建议数据结构示例：

```ts
type SqlExecutionArtifact = {
  id: string;
  messageId: string;
  index: number;
  sql: string;
  status: 'pending' | 'running' | 'success' | 'error';
  columns?: Array<{ name: string; type?: string }>;
  rows?: unknown[];
  rowCount?: number;
  error?: string;
  chartConfig?: unknown;
  createdAt: string;
};
```

#### 展示建议

- 在 SQL Artifact 中按执行顺序展示多个 SQL block；
- 每个 SQL block 下方展示对应结果表格、错误信息或 Chart；
- 支持折叠/展开，避免页面过长；
- 每个结果块提供复制 SQL、复制结果、重新执行等操作；
- 不应该用最后一条 SQL 覆盖前面的 SQL Artifact。

#### 验收标准

- 一次回复中包含 2 条及以上 SQL 时，每条 SQL 都能独立展示；
- 每条 SQL 的结果、错误、Chart 都不会互相覆盖；
- 历史对话重新打开后，多 SQL Artifact 的顺序和内容保持一致；
- 前端 state 和持久化数据中不再只保存单个 `sql` 或单个 `result` 字段。

### 3. 对话消息覆盖问题：AI 最后一条消息覆盖用户消息

#### 现象

对话过程中，AI 最后一条消息可能覆盖掉用户刚发送的消息，导致消息列表顺序或内容异常。表现可能包括：

- 用户消息发送后消失；
- AI 回复更新时覆盖了用户消息节点；
- 流式输出过程中最后一条 message 被错误更新；
- 消息 ID 不唯一，导致前端 diff/patch 时定位错误。

#### 影响

- 用户无法确认自己发送过什么问题；
- 对话上下文显示不可信；
- 持久化后历史记录可能缺失用户消息。

#### 建议排查方向

- 检查新增用户消息和新增 AI 消息时是否使用了不同且稳定的 `messageId`；
- 检查流式响应更新逻辑，AI token append 时只能更新 assistant message，不能更新最后一条任意 message；
- 检查 reducer/state update 是否通过 `role === 'assistant'` 和 `messageId` 双重定位；
- 检查 optimistic UI 插入用户消息后，服务端返回消息是否错误合并；
- 检查 key 是否使用数组 index，建议改为稳定 id。

#### 推荐修复原则

消息列表应遵循 append-only + targeted update 原则：

1. 用户发送消息时立即 append 一条 `role: 'user'` 消息；
2. AI 开始回复时 append 一条新的 `role: 'assistant'` 占位消息；
3. 流式输出只更新该 assistant 消息的 content；
4. 不通过 `messages[messages.length - 1]` 盲目覆盖；
5. 不使用数组下标作为消息唯一 key。

#### 验收标准

- 用户消息不会被 AI 回复覆盖；
- 连续发送多轮消息时，对话顺序稳定；
- 流式输出、失败重试、刷新页面后消息仍然正确；
- 每条消息都有稳定唯一 ID，并在 UI key、状态更新和持久化中一致使用。

### 4. 持久化对话记录缺少前端入口 / 展示不完整

#### 现象

后端或本地可能已经有对话持久化能力，但前端缺少完整入口或展示能力。用户无法方便地查看、恢复、管理历史对话。

#### 影响

- 用户刷新页面后难以恢复上下文；
- 无法从历史记录继续分析；
- SQL Artifact、Chart、用户消息、AI 消息的长期可用性不足。

#### 建议功能

前端建议补充历史对话管理能力：

- 左侧或顶部提供“历史对话”入口；
- 展示会话列表，包括标题、更新时间、消息数量等；
- 点击历史会话后恢复完整消息列表；
- 恢复 SQL Artifact、Chart 配置和执行结果；
- 支持删除会话；
- 支持重命名会话；
- 支持新建会话；
- 可选：支持搜索历史会话。

#### 建议数据结构

```ts
type Conversation = {
  id: string;
  title: string;
  createdAt: string;
  updatedAt: string;
  messages: Message[];
  artifacts?: SqlExecutionArtifact[];
};

type Message = {
  id: string;
  conversationId: string;
  role: 'user' | 'assistant' | 'system' | 'tool';
  content: string;
  createdAt: string;
  status?: 'sending' | 'streaming' | 'done' | 'error';
};
```

#### 验收标准

- 页面刷新后可以恢复当前会话；
- 历史会话列表可以正常展示；
- 打开历史会话后，用户消息、AI 消息、SQL Artifact、Chart 均能恢复；
- 删除、重命名、新建会话行为稳定；
- 前端状态与持久化数据一致。

### 5. 设置页面显示优化

#### 现象

设置页面存在显示或交互体验可优化点，可能包括布局拥挤、信息层级不清、表单控件对齐不一致、状态反馈不足等。

#### 优化建议

- 按配置类型分组，例如数据库连接、模型配置、主题设置、系统设置等；
- 使用统一的表单间距、label 宽度和控件宽度；
- 增加保存成功、保存失败、加载中状态；
- 对危险操作增加二次确认；
- 对必填项、格式错误、连接失败等情况提供明确提示；
- 移动端或窄屏下保持可读性；
- 对长文本、密钥、连接字符串等字段提供隐藏/显示切换。

#### 验收标准

- 设置项分组清晰，视觉层级明确；
- 表单控件对齐统一；
- 保存、失败、加载状态明确；
- 窄屏下无明显溢出或错位；
- 敏感配置不会默认明文暴露。

---

### 6. 对话页排版和显示优化

#### 现象

对话页整体排版和显示体验需要优化，尤其是消息、SQL block、结果表格、Chart、Artifact 区域的层级和可读性。

#### 优化建议

##### 消息展示

- 用户消息和 AI 消息使用清晰的视觉区分；
- 消息宽度限制在合理范围，避免长行影响阅读；
- 支持 Markdown、代码块、表格的稳定渲染；
- 流式输出时减少布局抖动；
- 错误消息要和普通 AI 回复区分。

##### SQL 展示

- SQL 代码块使用等宽字体；
- 支持语法高亮；
- 提供复制按钮；
- 多 SQL 场景按顺序编号，例如 `SQL 1`、`SQL 2`；
- SQL 与结果表格之间建立明确关联。

##### 结果表格

- 表头 sticky；
- 大数据量时分页或虚拟滚动；
- 空结果显示明确空状态；
- 错误结果显示错误原因；
- 列宽可读，长内容可截断并支持展开。

##### Chart 展示

- Chart 区域提供标题和数据来源说明；
- Chart 渲染失败时显示降级信息；
- Chart 与对应 SQL 结果保持绑定；
- 容器宽高自适应，避免压缩变形。

#### 验收标准

- 对话页在多轮消息、多 SQL、多 Chart 场景下仍然清晰；
- 长 SQL、长结果、长 AI 回复不会破坏布局；
- 代码块、表格、Chart、Artifact 的层级关系明确；
- 桌面端和窄屏端都能正常阅读和操作。

---

## 建议优先级

### P0

- 修复 AI 消息覆盖用户消息；
- 修复多 SQL 结果只保留最后一条 SQL 的问题。

### P1

- 修复 Chart 渲染与多结果绑定问题；
- 补齐历史对话前端入口与恢复能力。

### P2

- 设置页面显示优化；
- 对话页整体排版优化。

## 建议拆分任务

1. 修复消息列表状态更新逻辑，避免 AI 消息覆盖用户消息；
2. 重构 SQL Artifact 数据结构，支持多 SQL、多结果集；
3. 修复 Chart 与 SQL 结果的绑定关系；
4. 增加历史对话列表和会话恢复 UI；
5. 优化设置页布局和表单状态反馈；
6. 优化对话页 Markdown、SQL、表格、Chart 的展示样式。

## 回归测试用例

### 用例 1：单轮普通对话

1. 用户发送一条普通问题；
2. AI 返回普通文本；
3. 检查用户消息和 AI 消息均正常显示，顺序正确。

### 用例 2：流式回复

1. 用户发送问题；
2. AI 流式输出；
3. 检查用户消息不会被覆盖；
4. 检查 AI 内容只 append 到 assistant 消息。

### 用例 3：多 SQL 执行

输入示例：

```sql
SELECT * FROM users LIMIT 10;
SELECT COUNT(*) FROM users;
```

期望：

- 页面显示 `SQL 1` 和 `SQL 2`；
- 两条 SQL 都有独立执行结果；
- 任意一条失败时不影响另一条展示。

### 用例 4：多 SQL + Chart

1. AI 返回两条 SQL；
2. 每条 SQL 都生成可视化结果；
3. 检查每个 Chart 使用对应 SQL 的数据，而不是统一使用最后一个结果。

### 用例 5：刷新恢复历史会话

1. 创建一轮包含 SQL 和 Chart 的对话；
2. 刷新页面；
3. 从历史会话恢复；
4. 检查消息、SQL Artifact、结果表格、Chart 均存在且顺序正确。

## 备注

以上内容可作为一个总需求文档，也可以进一步拆分为多个 GitHub issue。建议先处理 P0 问题，确保消息和 SQL 结果不会丢失，再做 Chart、历史记录和 UI 优化。
